# AI Deal Creator — Claude Code Instructions

## Product purpose

This is an AI-assisted CLO deal structuring workspace. Claude's role is to
orchestrate workflows, interpret outputs, and draft language — not to compute
official structured finance numbers.

---

## Hard rules

These rules are non-negotiable and apply to every task.

### 1. Never invent official numbers
Do not calculate, estimate, or fabricate CLO outputs such as equity IRR,
tranche sizes, OC cushions, WAL, or any other metric that an official
cashflow engine is responsible for.

### 2. Deterministic numbers, generative language
- Numbers → come from the cashflow engine via MCP (`cashflow-engine-mcp`)
- Language → generated by Claude from approved sources
- These two must never be mixed or substituted for each other

### 3. Tag every significant output by source type
Every analyst-facing output must identify where each piece of information came from:

| Tag | Meaning |
|---|---|
| `[calculated]` | Produced by the official cashflow engine |
| `[retrieved]` | Fetched from a data source (pool tape, historical deal, docs repo) |
| `[generated]` | Drafted by Claude from grounded inputs |

### 4. External-facing content requires approval
- Any output destined for investors or external parties must pass compliance-review-agent
- No publish action without an approval record with `status = "approved"`
- Never auto-publish — always require explicit human release

### 5. Source grounding is mandatory
- Cite the scenario_id and run_id for all engine outputs
- Cite the doc_id or language_pack_id for all approved language used
- If a source is missing or stale, state that explicitly — do not fill the gap

### 6. Scenario actions are structured, not assumed
- Convert all natural-language deal requests into a typed scenario payload before execution
- Never submit a scenario without showing the structured parameters first
- Flag ambiguous or incomplete parameters — do not guess

## Architecture

```
.claude/agents/         # Subagent definitions — each agent has one responsibility
.claude/skills/         # Reusable Claude skills (/compare-scenarios, /draft-investor-summary, ...)
infra/hooks/            # Hook scripts — deterministic enforcement of policy
infra/mcp/              # Real MCP server definitions (Phase 2+)
mcp/                    # Mock MCP stubs (Phase 1 development only)
app/domain/             # Canonical domain models
app/schemas/            # Pydantic validation — run before any workflow
app/workflows/          # Workflow orchestration logic
app/services/           # Shared services (audit logger)
app/prompts/            # Prompt templates — kept separate from business logic
```

---

## Subagent delegation

Use these agents for their specific responsibilities. Do not collapse roles.

| Agent | When to delegate |
|---|---|
| `deal-schema-agent` | New deal creation, pool tape import, field normalization |
| `structure-analyst` | Tranche logic, parameter review, structural tradeoffs |
| `scenario-runner` | Engine submission, run status, output collection |
| `analytics-explainer-agent` | "Why did X change?", driver analysis, version comparison |
| `document-drafter` | Investor summary, IC memo, FAQ, change log |
| `compliance-reviewer` | Any output headed for external audiences |
| `deal-orchestrator` | Top-level intent routing — delegates to the above |

---

## Development workflow

Follow this order when making changes:

1. Inspect existing contracts first (`app/domain/`, `app/schemas/`)
2. Update schemas before changing workflow logic
3. Keep prompt templates in `app/prompts/` — never embed them in business logic
4. Write tests for all validation and comparison logic
5. Never mix mock data (`mcp/`) with production MCP connectors (`infra/mcp/`)
6. Every workflow must emit an audit event via `app/services/audit_logger.py`

---

## Domain vocabulary

Use these terms consistently in code, prompts, and docs:

- **deal** — the top-level transaction object
- **pool** — the collateral portfolio backing the deal
- **tranche** — a liability class with defined seniority, size, coupon
- **scenario** — a parameterized run submitted to the cashflow engine
- **run** — a single execution of a scenario in the engine (has a run_id)
- **investor summary** — external-facing deal narrative (requires approval)
- **committee memo** — internal IC-facing document (requires approval)
- **approval** — an explicit sign-off record gating publication
- **publish** — release of an artifact to an internal or external portal

---

## What Claude must not do

- Compute equity IRR, OC ratios, tranche NPV, or any official metric directly
- Auto-publish any material without an approval record
- Use mock engine outputs as if they were official engine outputs
- Embed deal data in prompt strings — pass it as structured parameters
- Skip audit logging for workflow actions
